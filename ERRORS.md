# ERRORS.md — Vanta failure log

Append-only. Approaches that took >2 attempts. Check this before debugging similar tasks.

## 2026-06-10 — Alt-screen TUI ghost frames (stacked composer boxes)

**Symptom:** With `VANTA_NO_FLICKER=1`, the TUI rendered the composer box repeatedly stacked down the screen (ghost frames), each with a varying-width top border. Worse on short terminals. Sometimes a blank screen.

**What failed (4 wrong attempts — all theorized from screenshots, none verified):**
1. SIGINT/SIGTERM/exit handlers to restore the terminal — addressed cleanup, not the ghosting.
2. Manual `\x1b[2J\x1b[H` clear before `render()` + a resize handler — caused a *blank* screen: clearing the screen out-of-band desynced Ink's internal `previousOutput`, so log-update computed a zero-diff and wrote nothing.
3. Swapped manual `\x1b[?1049h/l` escape codes for Ink v7's built-in `alternateScreen: true` option — correct API, but ghosts persisted.
4. Removed the 36-line banner from the alt-screen render path — reduced overflow but ghosts persisted.

**Root cause:** `app.tsx` read `const ALT_SCREEN = process.env.VANTA_NO_FLICKER === "1"` at **module-load time**. The `.env` is loaded at **runtime** (`cli.ts` `main()` → `loadEnv()`), long after the static `import` graph evaluated — so `ALT_SCREEN` was frozen `false`. Meanwhile `launch.tsx`'s `inAltScreen()` reads at runtime and correctly enabled Ink's alt-screen buffer. The two surfaces disagreed: Ink entered the alt buffer, but `App` rendered the `<Static>` + banner path (frozen `false`) **inside** it. `<Static>` has no scrollback and the banner overflows, so every render tick ghost-stacked.

**What worked:** Thread `altScreen` as a **prop** from `launch.tsx` (runtime-correct) into `App`. The two surfaces now agree by construction. Verified in tmux at 102×27 (the failing size): clean single composer, no banner, no ghosts, survives resize; normal mode still renders the banner via `<Static>`.

**Lesson:** Any `const X = process.env.Y` at module top-level in a file imported by `cli.ts` is frozen *before* `loadEnv()` runs — it will always read the default. Read env at runtime (in a function, or threaded as a prop), never at module load. `THEME`/`VIM_ENABLED`/`SPINNER` in `app.tsx` have the same latent bug.

**Meta-lesson:** I edited 4 times from screenshots without once capturing the actual terminal bytes or driving the real app. The fix came only after reading the env-load order and verifying in tmux. Reproduce + verify before editing, not after.

## 2026-06-10 — Alt-screen ghost frames pt 2: resize storms (the prop fix wasn't the whole story)

**Symptom:** With the altScreen-prop fix in (entry above), a single resize looked clean — but an ANIMATED macOS drag (a storm of SIGWINCH) still ghost-stacked composer boxes. Reproduced headlessly: tmux 100×29 + a tight `tmux resize-window` width loop (grow AND shrink) → 3 stacked boxes after 4 rounds. One clean resize proves nothing.

**Root cause (read from Ink 7.0.5 source, then confirmed live):** short frames render via log-update's RELATIVE path — `eraseLines(previousLineCount)` anchored at the cursor (`ink/build/log-update.js`). Ink only full-clears when a frame overflows the viewport (`shouldClearTerminalForFrame`, ink.js) and its resize handler only clears on width DECREASE. Vanta's ~8-line frame never overflowed → every mid-storm width increase under-erased rewrapped lines → old frame tops survived → stacking.

**What worked:** make every frame overflow on purpose. `tui/alt-frame.tsx` renders a full viewport of blank filler ABOVE the content, so outputHeight always exceeds rows → Ink takes its clearTerminal path (absolute, home-anchored full rewrite) on every frame, and the write's final scroll leaves the content bottom-anchored with blanks above. Reactive rows/cols (`useTermSize`) re-render on resize; a debounced redraw nonce (`useResizeRedraw`) forces one post-storm commit as backstop. Banner returned as scrollable VirtualTranscript entries (`bannerEntries`), safe under the new regime.

**Dead ends inside this fix (caught by ink-testing-library probes before shipping):**
- `justifyContent="flex-end"` + `overflow="hidden"` does NOT bottom-anchor overflowing content — Yoga clamps children to the top edge, clipping the NEWEST lines.
- `flexDirection="column-reverse"` — same clamp; children overdraw each other at the top.
- A trailing-space redraw nonce is dead: Ink `trimEnd()`s every output line (`ink/build/output.js`) so the frame string never changes. Use a non-whitespace glyph.
- A fixed-height frame needs `flexShrink={0}` on every fixed-size child or Yoga silently collapses them.

**Lesson:** verify resize handling with a rapid storm in BOTH directions, never a single resize. And never build on an assumed Yoga layout behavior — probe it with a 10-line ink-testing-library render first.

## 2026-06-11 — Killed the kernel under a live session → bricked transcript (Codex 400)

**Symptom:** After rebuilding the kernel I killed the old process (port 7788) while a Vanta session was OPEN. The live session's next tool call failed (`error: fetch failed` — safety client lost the kernel), the turn aborted mid-dispatch, and from then on EVERY turn failed: `Codex request failed (400): No tool output found for function call call_…`. The session was bricked; /compact etc. appeared "broken" too (same kernel-down root cause).

**What failed:** (1) `applySafetyGate` had no try/catch around `safety.assess()` — a kernel fetch failure threw through dispatchTool → runTurn → the turn died AFTER the assistant tool_calls message was pushed but BEFORE any tool result landed. (2) `sanitizeMessages` only dropped orphaned tool RESULTS, not dangling assistant tool CALLS — so the malformed transcript replayed forever (saved sessions resume it too).

**What worked:** Fail closed, gracefully: assess() failure now returns a blocked tool result ("safety kernel unreachable — restart vanta") so the transcript stays consistent and the session survives; approval-queue bookkeeping is best-effort. And `sanitizeMessages` now synthesizes a stub result for any dangling call ("[no result — the turn was interrupted…]") right after its assistant message — self-heals new aborts AND existing bricked session files on their next call.

**Lesson:** the kernel is a live dependency of every open session — never kill/replace it while a session is running without restarting the session; and any code path that appends an assistant tool_calls message must guarantee a result lands for every call id, even on abort. Transcript consistency is an invariant, not a happy-path property.

## 2026-06-13 — TUI v2 streaming duplication (ghosting regression)
What failed: the live region rendered the full growing assistant stream (`⏺ {streaming}`). Once a turn's <Static> transcript + the streamed text exceeded the terminal height, Ink could no longer erase the prior dynamic frame, so every token re-render stacked a copy (~70 duplicate lines on screen) — the exact ghosting the Claude-method rebuild was meant to remove.
What worked: bound the non-<Static> region. Stream shows only a clipped TAIL (last 6 lines, width-clipped so no wrapping — `ui/stream-view.tsx streamingTail`); todo panel capped to 6 rows (`ui/todo-panel.tsx`); busy spinner hidden while text is flowing. The complete response still commits to <Static> scrollback intact on turnEnd.
Why it failed: Ink re-renders the non-Static region in place and can only do so while that region fits the viewport. An unbounded live region violates that invariant — the fix is to keep the live region small and let <Static> + the terminal own everything else.

## 2026-06-13 — Ghosting, second vector: streamed text held through a tool/approval phase
What failed: bounding the stream tail (above) fixed pure streaming, but the reducer only committed streamed text on `turnEnd`. When the model streamed a long preamble and THEN called a tool — especially an approval-gated one (`write_file` to `~/Desktop`, outside the repo root → kernel `ask`) — the full text sat in the redrawing live region for the entire tool+approval window. Each redraw of that tall region leaked a copy into scrollback (~40 stacked copies of the "Goal: …" preamble).
What worked: commit assistant text to <Static> the instant a tool call follows it. New reducer `commitText()` flushes `streaming` → an assistant entry on `toolCall` (and via `flush()` before any user/thinking/note entry), so the text leaves the live region before the tool runs. During the tool/approval phase the live region is now just the spinner + active-tool line. `commitStreaming` (turnEnd) reuses `commitText`, so trailing text still commits and never duplicates.
Why it failed: the live-region-size invariant isn't only about streaming length — ANY committable content (text) left in the dynamic region during a long-held phase (tool exec, approval HITL) will ghost. Rule: text and tools both leave the live region for <Static> as soon as the next thing happens, not at turn end.

## 2026-06-15 — Ghosting, third vector: Footer height unstable when goal is null
**Symptom:** ~9-10 stacked copies of the Composer placeholder on startup. Screenshot shows every second a new ghost copy accumulates. `{` artifacts on right edge = right border of the rounded Composer box.
**Root cause:** `Footer` conditionally rendered the goal line: `{props.goal ? <Text>◇ goal</Text> : null}`. When `goal=null` (before async `useActiveGoal` resolves), Footer was 2 lines. When goal loaded, it grew to 3 lines. `useClock` (1000ms interval) triggered a re-render each second. Ink erased `previousLineCount=2` lines but the terminal had 3, leaving 1 unclaimed row per second.
**Fix 1 tried:** Wrapped null case in always-present `<Text>` but used `""` for null value — still 0 lines in Ink because Yoga collapses empty-string text nodes. `<Text dimColor>{""}</Text>` renders as a blank row in Ink — i.e., the `\n` IS emitted but the content is empty. Contrast: the line IS present physically.
**What worked:** Changed `""` to `" "` (single space). In Ink/Yoga, a space keeps the text node at height=1 (Yoga allocates 1 row for a 1-character text node). Footer is now always 3 lines regardless of goal.
**Test:** `Footer` with `goal=null` must produce 3 `\n` characters. Use `(output.match(/\n/g) ?? []).length` — NOT `split("\n").filter(l => l.length > 0)` which drops the blank goal row. Test: `render.test.tsx` "Footer height stable".
**Lesson:** Ink/Yoga collapses `<Text>{""}</Text>` to 0 physical rows. Any conditional `null` or `""` in a height-critical position causes ghosting when a periodic timer re-renders. Use `" "` (space) to hold a line's height. Test with newline count, not non-empty-line filter.

## 2026-06-15 — Ghosting, FOURTH vector (the real one): resize storms in the new inline renderer
**Symptom:** ~6-7 stacked Composer boxes after RESIZING the terminal window (not idle). The screenshots that triggered the Footer fix above were actually this — the user clarified "it happens when I resize." The Footer fix was real but addressed a different (lesser) vector.
**Root cause:** The June-13 rebuild to inline Ink + `<Static>` (`src/ui/`) deleted the old alt-screen renderer AND its resize protection (`tui/alt-frame.tsx`, see the 2026-06-10 pt-2 entry). Ink 7.0.6's `Ink.resized` (ink/build/ink.js) clears ONLY on width *decrease*, and even then via log-update's RELATIVE erase `eraseLines(previousLineCount)` — a LOGICAL line count. A terminal physically REWRAPS already-displayed lines when width changes, so that count is wrong in BOTH directions. The Composer's full-width rounded border is the guaranteed culprit: at width W its border is W chars; shrink below W and the terminal wraps it to 2 physical rows, but Ink erases 1 → the border top survives → stacks on every resize.
**Reproduction (ground truth, not screenshots):** `scripts/ghost-storm.sh` — tmux at 100×30 running `scripts/ghost-repro.tsx` (the real components in app.tsx's idle tree), a rapid grow/shrink width loop, then `tmux capture-pane` and count "Ask Vanta" occurrences. Before fix: 7. After: 1. (Per the pt-2 meta-lesson: storm BOTH directions, never a single resize; verify in a real terminal emulator, never from a screenshot.)
**Dead ends (confirmed, do not retry):** (1) App-level resize listener that writes `\x1b[2J`/clearTerminal out-of-band — desyncs Ink's log-update cursor/line state (same as 2026-06-10 attempt #2); next render diffs wrong → blank or worse ghosts. (2) Keeping lines under terminal width — impossible; the box border is always full-width, and the terminal rewraps content that was displayed at the OLD width regardless of the new layout. (3) Going fullscreen so Ink always full-clears (the old alt-frame trick) — works but forfeits native `<Static>` scrollback, the whole point of the inline rebuild.
**What worked:** `src/term/resize-fix.ts` — reach Ink's live instance via its internal WeakMap (`ink/build/instances.js`, loaded by direct file:// path since Ink's `exports` map blocks the subpath) and APPEND a resize listener (don't replace Ink's — use-window-size has its own). On resize: `calculateLayout()`, set `ink.lastOutputHeight = Number.MAX_SAFE_INTEGER`, then `onRender()`. The huge height makes Ink's own `shouldClearTerminalForFrame()` see `wasOverflowing` → it takes the absolute-clear path (`clearTerminal + fullStaticOutput + output`, immune to rewrapping) and its `log.sync()` re-syncs cursor state + restores the real height. Wired in `ui/launch.tsx` after `render()`. Guarded: non-TTY or changed Ink internals → silent no-op (default behavior). Tests: `src/term/resize-fix.test.ts` (mechanism) + `scripts/ghost-storm.sh` (integration, needs tmux).
**Lesson:** Inline Ink's relative erase is fundamentally unsafe across width changes — any full-width element ghosts on resize. The only correct fix is to force Ink's absolute-clear path; the cleanest lever is `lastOutputHeight` (drives `shouldClearTerminalForFrame`). And: when a screenshot looks like vector A (idle timer) but a fix doesn't hold, ASK what action triggers it before fixing again — "it happens when I resize" reframed the whole diagnosis.

## 2026-06-20 — `hooks/file-watch.test.ts` flakes under full-suite load (ENOENT on the marker)
**Symptom:** `FileChanged hook watcher > fires for matching changed files` fails ~1-in-N full-suite runs with `ENOENT … access('…/file-hooked')` after the full 750ms `waitFor` budget; passes 2/2 every time in isolation.
**Root cause:** `fs.watch(root, {recursive:true})` (FSEvents on macOS) arms ASYNCHRONOUSLY and exposes no readiness signal — `startHookFileWatcher` returns before the OS watch is live. The test did one `writeFile(watched.txt)` immediately after, then polled for the marker. Under full-suite CPU load the arming slips past that single write, so the change event is **MISSED, not delayed** — the marker is never created, and any amount of extra waiting can't recover a lost event. That's why a longer timeout (the obvious "fix") would not have worked.
**Dead end (do not retry):** bumping the `waitFor` retry count / sleep — the event is lost, not slow; more polling just fails slower. Also rejected: adding a "settle" delay inside `startHookFileWatcher` — `fs.watch` has no awaitable ready state, so a sleep is cargo-cult and still races under heavier load.
**What worked:** re-fire the trigger INSIDE the poll loop — `waitForMarker(marker, trigger)` calls `trigger()` (rewrite `watched.txt` with changing content) every 25ms until the marker appears. Even if the first writes land during arming, a later one is guaranteed to land after the watch is live and is caught within one 50ms debounce. Deterministic for any realistic arming latency (80×25ms budget). Production needs no change: the watcher is started once at session start and files change seconds-to-minutes later, well after arming.
**Why it failed originally:** the test encoded a happy-path assumption (watcher is live the instant `startHookFileWatcher` resolves) that isn't true for `fs.watch`. Lesson: any `fs.watch`-based test must re-trigger the change until observed, never trigger once and wait.
**Bonus trap found same session:** running `npx vitest run` from the REPO ROOT (cwd had drifted) uses the root vitest config, which also scans the bundled `reference/` quarry repos (goose, hermes-agent) → ~159 spurious file failures that look catastrophic but aren't Vanta's. Always run the TS suite from `vanta-ts/`; confirm `pwd` first (shell cwd persists across Bash calls).

## 2026-06-20 — Ink UI tests flake under full-suite load (assert after a fixed tick count)
**Symptom:** `ui/mcp-panel.test.tsx` (and the same pattern latent in `config-panel`, `sandbox-panel`, `trust-dialog`, `composer`) fails intermittently only under full-suite load: a frame assertion after a keypress sees the PRE-keypress view (`expected 'MCP servers (2)…' to contain 'github · tools (1)'`). Passes every time in isolation.
**Root cause:** the tests did `inst.input(x); await ticks(n); expect(lastFrame()).toContain(...)`. A keypress → `useInput` → `setState` → Ink repaint takes a nondeterministic number of flush cycles; under load a fixed `n`-tick wait races the repaint and the captured (cumulative) frame is stale. Same class as the file-watch flake: a fixed wait against an async event.
**What worked:** poll-until-condition helpers in `test-render.tsx` — `waitForFrame(inst, text)` (ticks until the cumulative frame contains `text`) for frame assertions, `waitUntil(() => spy.mock.calls.length > 0)` for callback/state assertions. Both return as soon as the condition holds (fast) and throw on timeout. Converted the 5 files; kept fixed waits only for negative assertions (`not.toHaveBeenCalled`) and intermediate keypress sequencing. Documented the rule in `ui/CLAUDE.md`.
**Dead end / real bug caught:** my first `waitForFrame` target for the tool-DETAIL test was `"Create a GitHub issue"` — but the tool DESCRIPTION also renders in the tools-LIST view, so the poll returned after the first ⏎ (list), before the detail view rendered, and `"Schema"` was missing. **Consistent failure across runs = a real bug, not a flake.** Fix: wait on a marker UNIQUE to the target view (`"Schema"` is detail-only), then assert the rest. Lesson: a poll wait-target must be unique to the post-input view, or it resolves on the wrong frame.
**Same family swept across the whole suite (not just the 3 that happened to fail):** the underlying anti-pattern is "fixed sleep/tick, then assert a POSITIVE async result." Audited every `await new Promise(setTimeout)` in tests and converted the genuine cases to poll-until-condition with a local `waitUntil`: `ui/config-actions` (fire-and-forget settings write → poll the host-publish signal), `ui/use-submit` (poll the send/dispatch spy), `uds/peers` (poll the delivered `received` array), `agent/permission-gate` (poll the hook-written marker content), `mcp/client` (poll the `seen` events). Left LEGITIMATE fixed delays alone: negative assertions (`not.toHaveBeenCalled` after a bounded wait), recovery polls that re-check (`runtime-events`, `notify` — no missed-event risk because they run `touch` directly, unlike the FSEvents watcher above), mock-internal delays (`plan-v2`), timestamp-gap forcing (`radar/promote`), and interval/debounce timing tests (`use-blink`). Result: 2 consecutive fully-green full-suite runs (606/606 files, 5077/5077 tests).
