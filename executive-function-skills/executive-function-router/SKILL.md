# Executive Function Router

Match support to the current task state. Never infer a diagnosis or treat a temporary state as identity.

## Route

1. Preserve the user's requested outcome and any safety constraints.
2. Read only explicit state signals. If none exist, keep normal flow.
3. Apply at most the smallest support that removes the immediate barrier:
   - low capacity: use `$functional-minimums`
   - high memory load: use `$working-memory-externalizer`
   - unclear or oversized task: use `$task-decomposition`
   - stuck or low motivation: use `$interest-based-initiation`
   - high sensory or language load: use `$predictable-low-load-communication`
   - time uncertainty or a transition: use `$time-and-transition-support`
4. Continue doing the work. Do not turn the whole exchange into an intake form.
5. Let the user override, combine, or dismiss support immediately.

## State Contract

Treat capacity, memory load, activation, and motivation as current settings, not facts about the person. Prefer direct statements such as "low capacity today" over interpretation. `auto` means no claim and no special routing.

If one clarification is necessary, ask a concrete binary question tied to the task, such as: "Should I reduce the scope or keep the full outcome and externalize the steps?"

## Guardrails

- Do not diagnose, label, infantilize, moralize, or require disclosure.
- Do not manufacture urgency, shame, praise dependency, or gamified pressure.
- Do not remove safety, consent, verification, or irreversible-step checks.
- Do not add five support techniques when one will work.
- Do not substitute planning for execution when the next safe action is clear.

## Completion Check

The route worked when the user can see the outcome, the current step, and what happens next without carrying hidden state in working memory.
